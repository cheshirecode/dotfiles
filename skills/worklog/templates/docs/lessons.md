# Worklog lessons

Ledger of *cross-task* lessons surfaced by shipped work. **Cross-task** means the insight applies beyond the single task that produced it — a pattern observed across multiple tasks, a recurring failure mode, or a constraint future readers shouldn't have to re-derive from `git log`. One-task surprises that don't generalize stay in the task body, not here.

The signal: "we'd want a future agent or human to know this without re-reading 10 commits."

## Quickref — highest-recurrence (read this; skip the rest for non-review modes)

- **YAML colon quoting (B5):** Quote any frontmatter value containing a colon — `next_action: "Next: foo"` not bare.
- **`related[]` not body prose (H4):** Cross-task slug mentions → `related[]` immediately; `lint --cross-task` and `bin/children.sh` don't see body text.
- **Doc-code atomicity (G1/G2/B6):** Script change that AGENTS.md / cheatsheet / helpers.md describes → update the doc in the same commit.
- **`git log -S` before fixing a 0-metric (A1):** Verify the event wasn't intentionally removed before drafting an instrumentation fix.
- **Checkpoint staged-guard (c76b361):** `bin/checkpoint.sh` refuses staged paths outside the target file — if a flip commit has a large diff, investigate before force-pushing.

Each entry: short title (the lesson, one phrase) + 1–3 sentence rationale + `(short-sha, item-id)` reference back to the commit that proved it.

Newest at top within each month section. **Entries are mutable** — refine wording, correct mistakes, expand rationale, or supersede with a clearer formulation as understanding deepens. To refine: edit the entry in place and mention which entry was touched in the commit body (e.g., `lessons: refine B5 entry — clarify which scripts must quote`). Git history is the audit trail of how each lesson evolved; the file itself reflects current best understanding.

## 2026-06

- **Build-and-run beats static reasoning (and a multi-agent council) for runtime-flow bugs — verify the actual launch path before locking in a mechanism.** Diagnosing the iOS in-app "shows marketing landing instead of login" bug, both a 4-angle council and careful static analysis confidently settled on "logged-out webview loads apex `/` anon → marketing." A 2-minute `Ideogram-Production` sim build disproved it: `RootView` gates `signedOut → native LoginView` (since Jan 2024), so the webview only loads when `signedIn`. The real mechanism was subtler — the app never injects the web `session_cookie` into the webview (no `WKHTTPCookieStore` use), so the webview's *first* load of `/` is cookie-less, and PR #12579 (apex gate, same day) turned that cookie-less `/` into a marketing redirect → a fresh-install-only bug. The fixes happened to be right, but the premise was mechanically wrong. Lesson: when the bug is about runtime/auth/session flow, spend the few minutes to build+run before settling the "why" — councils and greps describe the code, not the live state. ([[ios-webview-landing-page-skip]])

- **Autosave and checkpoint are two different verbs, not one with a bypass.** Autosave is the **durability layer** (slugless safety snapshot on PreCompact / SessionEnd / manual call); checkpoint is the **audit-trail layer** (per-slug logical-state-advance with `Worklog-Status:` trailers). The 20-40 autosave commits/month in this repo aren't noise — they're the cross-agent portability cost that lets Codex / Cursor / shell sessions never lose work when Claude's hook ecosystem isn't available. Surfaced 2026-06 by the protocol-review council (Stage 3 conflict A5 vs B4). Resolution: emit `Worklog-Trigger: pre-compact | session-end | manual` on autosave commits so `git log` can filter; `bin/status.sh` already excludes autosave from standup output by default (`--include-meta` re-includes). Documenting this layering in AGENTS.md § Checkpoint discipline closes the "autosave is a defect" misread without adding a staged-path guard that would break the cross-agent escape hatch. ([[bin/autosave.sh]], [[bin/install-hooks.sh]])

- **Derived caches under `.cache/` are per-machine by design — that's the canonical KM/vault pattern, not a coverage gap.** Every cache file (`compact-kernels.{md,json}`, `index.jsonl`, `index.embeddings.jsonl`, `claims/`, `sessions/`) is regenerable from the git-synced vault; cross-machine drift is structurally impossible because each clone builds its own caches. Atomic-write idiom (write-temp + `mv -f`) prevents same-machine concurrent-session corruption. Documented in AGENTS.md § Derived caches 2026-06. Equivalent to Obsidian's local-index pattern. ([[AGENTS.md]])

- **Prod `ideogram.ai` smoke via headless curl gets CF-bot-challenged (`403 cf-mitigated: challenge`) on any path NOT in the Super-Bot-Fight-Mode skip rule, after a few requests — a FALSE 403, not a regression.** Bit me twice (2026-06, apex-gate launch): `/api/billing` + `/t/explore` showed 403 mid prod smoke and looked like the gate broke routing; they were fine. Verify true app health **in-cluster** to bypass CF entirely: `kubectl exec <frontend-pod> -c frontend -- curl -s -o /dev/null -w '%{http_code}' -H 'Host: ideogram.ai' http://localhost:8080/<path>`. Also: a browser surfaces a challenged *sub-resource* (e.g. `/_astro/*.js`) as a console **404** while curl sees **403** — same root cause (the bot challenge), not a missing file. Skip-ruled paths (no challenge) as of the launch: `/`, `/models/`, `/_astro/`; the other marketing paths needed adding. Don't trust a headless prod 403/404 — confirm in-cluster or in a real browser.

- **`bin/lint.sh --fix-related` must match the existing block's indent style — both indent-0 (`- slug:` at column 0) and indent-2 (`  - slug:`) are valid YAML for a parent mapping's sequence value, but they MUST NOT MIX in one block or the parser errors at the indent boundary.** Auto-stub always emitted indent-2; appending to an indent-0 block produced mixed indent → YAML parse error → blocks subsequent lint runs. Hit on `contact-sales-form-openapi-spanner` 2026-05-29. Fixed by detecting the first existing `- slug:` line's leading whitespace and using that indent for both new sequence items and their `note:` continuation. New blocks (no existing entries) still default to corpus-canonical indent-2. Block-end detection also rewritten — old logic treated any column-0 line as block-end, which failed for indent-0 sequence-item style (the `- slug:` lines themselves are at column 0). New logic: a line is inside the block iff it's a sequence item at the detected indent, a continuation at strictly deeper indent, or blank. Anything else (a sibling key at the same indent depth as `related:`) marks block-end. (commit landing this, three synthetic cases verified: indent-0, indent-2, no-existing-block.)

- **`bin/checkpoint.sh --status=X` must emit `Worklog-Status:` trailer whenever user explicitly passes `--status=`, not only when frontmatter actually changes.** Previously the trailer fired only on `NEW_STATUS != OLD_STATUS`, so `--status=in-progress` was a silent no-op when frontmatter already read `in-progress` — even though the lint divergence warning (`bin/lint.sh --cross-task`) explicitly asks for a `--status=X` invocation to realign trailer-vs-frontmatter. Result: workaround required bouncing status through `draft` to force a trailer-emitting transition (hit on `contact-sales-form-openapi-spanner` 2026-05-29). Fixed by changing the condition from "frontmatter status changed" to "user passed --status= explicitly" — the trailer reflects user assertion, not file diff. Noop-detect guard still short-circuits commits with zero semantic diff, so this doesn't introduce empty-commit noise; it only matters when a real commit is happening alongside the status assertion. (commit landing this; see also matching sub() multi-line scalar fix here mirroring bin/archive.sh.)

- **`bin/archive.sh`'s `sub()` regex must consume multi-line YAML scalar continuations, not just line 1.** The old regex `^{field}:.*$` matched a single line, so archiving a task whose `next_action:` was a quoted multi-line scalar (`'line1\n  line2\n  line3'`) replaced only the first line with `—` and left orphan continuation lines behind. Result: YAML parse error on the archived file, blocking later checkpoints across the corpus until manually fixed (`bin/lint.sh --cross-task` exit 1, pre-commit refuses every subsequent archive on the same machine). Fixed by extending the regex to `^{field}:.*(?:\n[ \t]+.*)*$` so continuation lines (any leading-whitespace lines immediately following) get consumed as part of the match and replaced cleanly. Hit twice in a single session (pipeline-comparison-pr1-review on 2026-05-29, then a synthetic repro to verify). Generalizes to any worklog helper that does per-field YAML rewrites — single-line regex is wrong wherever the field value can be a multi-line scalar (today: `next_action`, `summary` if multi-paragraph). ([[bin/archive.sh]])

- **For component-owned styling, theme reactivity comes from `getColor()` inside `styled()`'s callback form — not from threading `useNovaColor()` values through `$prop` or `sx`.** PR #12302 set out to fix "module-level `getColor()` is frozen" by migrating to `useNovaColor` + `$background` styled-prop plumbing. Reviewer (sarv19) flagged the prop plumbing; bugbot flagged a follow-up where switching `styled(Box)(() => ({...}))` → `styled(Box)({...})` silently froze `getColor()` reads at module load (`darkMode` is a mutable module-level `let`; only the callback form re-reads on render). Canonical pattern is documented in `frontend/react/rules/frontend-style-guide.md` (lines 12–15) and matches the bad/good code blocks in that guide. `getNovaColor()` is for React component bodies (Nova component props, icon `color=`, runtime values) — not for component-owned `styled()` definitions. Quick rule: if you're inside a `styled()`, use `getColor()` in callback form; if you're in a component body, `useNovaColor()` is fine for prop values but don't thread them back into a `styled()` via custom `$prop` indirection. **Recurrence signal:** the same `$background: string` anti-pattern still exists in `ActionBar.tsx` share-Popover slotProps and likely other gallery surfaces — sweep candidate. ([[gallery-additional-details-nova-tokens]], PR #12302 review thread `discussion_r3313431481`)

- **Tests that clone the repo MUST `rm -rf people/` before seeding their own LDAP, or production stale claims and corpus size leak into reap-walks and flake timing-sensitive assertions.** `tests/project/test_phase2.sh` flaked 6/8 — 4/10 (and 2026-05-15 review: "6/8 pass, 2/8 fail with reap cleared 3 claim(s)"). Two compounding causes: (1) production held a real orphan `claim:` on `pipeline-comparison-pr2-review.md` since 2026-05-15 by a `machine:UUID` Codex session, never released — the cloned scratch inherited it; (2) with ~130 <ldap> files in the test's `people/`, `bin/project.sh reap`'s per-file `_claim.py read` + `is-stale` subprocesses took ~13 s — long enough for the test's `stale_after=10s` window to expire on the freshly-ticked `p2-b` between heartbeat-tick and the is-stale check, falsely flagging it stale. Fix mirrors the `test_phase3.sh` fix shipped 2026-05-13: `rm -rf people` after clone, before seeding testuser. 10-run sweep: 6/8 → 10/10. **Generalizes: any test that exercises corpus-walking scripts (`lint`, `reap`, `verify --all`, `audit`, `status`) must strip the cloned corpus first** — both content drift AND iteration-overhead can flake the assertion. ([[worklog-review-2026-05-15]] Tier-2 carry-over)

- **When collapsing N independent things into 1 shared dispatcher/scheduler, name the property that was implicit before and either preserve it explicitly or document why sharing it is acceptable.** A shared rAF dispatcher (#11923) traded N independent failure domains for 1 — browsers swallow rAF callback throws and silently un-reschedule that one chain, so per-instance loops were implicitly fault-isolated. Collapsing silently shared the fault domain; one throwing tick → loop wedged for the whole page. Four follow-up commits (#11938) hardened it against throws, in-tick self-unsubscribe, in-tick re-subscribe, and Vite HMR re-eval — each catch was post-merge by a reviewer (human, Graphite Bugbot, Cursor Bugbot, human again). The class is: for any loop calling user-provided callbacks, default to per-iteration `try/catch`; for any module-scoped singleton state, pin on `globalThis` to survive HMR (the grain-texture cache in `runtime.ts` already did this — the dispatcher should have matched on first commit). Perf-only verification (CPU %, FPS, idle %) is symptom-driven and blind to latent failure modes; fault-inject before shipping shared infrastructure. (PR #11923 / [[gradient-loader-lag-regression]], PR #11938 / [[gradient-loop-shared-raf-isolation]])

- **Static investigation can't establish "X is the dominant cost" for a perf complaint — a DevTools / Playwright trace can.** For the `/t/explore` "loading animation lags at >50 images" report, static reading fingered `GradientLoopScene` (per-tile rAF + ~5 MB grain textures × N) as the prime suspect and shipped a defensive fix (#11822: shared grain + viewport gating). A subsequent DevTools trace (5.5 s on staging) and console probe (30 s on logged-in prod) showed the real picture: `ImageDecodeTask` 1175 ms / 39 tasks (~20% of trace time), heap delta ~3.7 MB per new tile dominated by `<img>` decode of `@2k` JPEGs; `FireAnimationFrame` only 207 ms / 2354 fires. The gradient-loop PR is still correct hygiene, but the dominant fix was a one-line src-ladder change to `@1k` thumbnails (#11824). For any perf complaint, capture a baseline trace before shaping a hypothesis — otherwise the first PR is the wrong PR even when its code is right. (PR #11822 / [[ui-gradient-loop-viewport-gate]], PR #11824 / [[ui-explore-thumbnail-1k]])

## 2026-04

- **A 0-event analytics query doesn't mean "instrumentation regressed" — verify with `git log -S 'EVENT_NAME'` before declaring a regression class.** When the SPA Web Vitals investigation found three Amplitude events at 0 over 7 d, I framed all three as one observability regression. Code search showed two were **intentional removals** months earlier (PRs #6012/#7412 retired `IMAGE_LOADED_LATENCY` with the dead `RemoteImage` component; PR #7603 dropped the canvas SUCCESS listener while keeping ERROR). Only the third (`[Amplitude] Web Vitals` on SPA paths) was a real bug — and that one needed an SDK-config fix, not the rewiring I had drafted. Cheap to falsify; expensive to fix something that's already shipped as removed. Generalizes: any "metric X is 0" finding in any observability tool warrants `git log -S` (or equivalent code archaeology) before action. (`enable-spa-web-vitals`, PR #11471)

- **`bin/checkpoint.sh` autostash-pulled uncommitted code into status-flip commits, silently bundling unrelated work.** Observed 3-4× in the 2026-04-28 session: a "worklog-foo: status → in-review" commit's diff contained 70 lines of new feature code from the working tree because `git pull --no-rebase --autostash` stashed-pulled-popped, and the subsequent `git add "$FILE"` plus the implicit "anything already in the index gets committed" behavior swept in any pre-staged changes. The audit trail looked like a small frontmatter flip; the diff was a feature ship. Mitigation in `bin/checkpoint.sh` (worklog-checkpoint-staged-guard, `c76b361`): refuse to commit when staged paths fall outside `<task-file>` + `--include` + rename source. Bypass via `WORKLOG_CHECKPOINT_FORCE=1` for intentional multi-purpose commits. Lesson generalizes: any single-purpose script that runs `git commit` after touching one path needs to assert nothing else is staged, or the audit trail rots silently. (`c76b361`)

- **`bin/sql.sh` (via `bq` CLI) is the escape hatch when the BigQuery MCP times out on Spanner-federated tables.** Queries against `prod_spanner_multi_region.*` tables are federated through Spanner; the BigQuery MCP repeatedly timed out / 502'd on `sampling_response` width/height percentile aggregations, even at 1-hour windows with shard sampling. The exact same SQL executed via `bin/sql.sh` (which shells out to `bq` directly) returned in ~10 s and cached the JSON for citation. When designing data-supported worklog tasks, save federated queries as `queries/<slug>/<name>.sql` from the start — the MCP path is fine for BQ-native datasets (`customer_360`, `sampling_request_stats`) but unreliable for Spanner federation. (`responsive-image-pipeline-design` § Data sources & evidence)

- **`bin/e2e-docker.sh` catches BSD-vs-GNU coreutils drift that local Mac development doesn't.** The 2026-04-27 LDAP-cache fix to `_lib.sh` passed on macOS but errored on debian/alpine because `stat -f %m` succeeds on Linux (showing filesystem info, not mtime), feeding non-numeric output into arithmetic and triggering a confusing `set -u` "unbound variable" against the token name. Local-only testing missed it; e2e on both images caught it in 3s. Run `bin/e2e-docker.sh` after any shell change involving `stat` / `date` / `sed` / coreutils flag forms — these are the dominant portability footguns. (`1f81125`)

- **`resolve_ldap`'s `gcloud config get-value account` call is ~1.2s on a warm shell — cache it.** Any helper that sources `bin/_lib.sh` and feels slow is almost certainly bottlenecked here, not in its own logic. `bin/status.sh` was 1080ms total; caching the resolved LDAP in `$TMPDIR/worklog-ldap-$USER` for 24h dropped it to 119ms. Before profiling helper internals, confirm whether `resolve_ldap` is on the path — that's the first cut. (`529b749`)

- **Tests that exercise scripts which write to user-owned caches must isolate `$TMPDIR` per-run.** First version of `tests/compact_kernels/test_kernels.sh` seeded a deterministic LDAP by writing to `$TMPDIR/worklog-ldap-$USER` directly — poisoning the real user's cache for 24h. Pattern: `export TMPDIR="$SCRATCH_ROOT/tmp"; mkdir -p "$TMPDIR"` before any helper that reads/writes shared cache files. Generalizes to any future test touching scripts that cache state in `$TMPDIR` or `~/.cache/`. (`b161f43`)

- **Hydrate Claude's `TaskCreate` / `TaskUpdate` / `TaskList` from `## Next` checkboxes on every multi-step task; the system reminder is the protocol guard, not a nag.** AGENTS.md § In-session progress visibility mandates this. The 2026-04-27 review session ran ~30 multi-step commits without a single `TaskCreate` invocation despite the reminder firing repeatedly. Self-evidence in `worklog-review-2026-04` Tier-1 #5. The convention is documented and the tool exists; the gap was pure agent compliance. On any session resume, read the active task's `## Next`, recreate `TaskCreate` entries for each unchecked item, then mark them as work flows. Skip only for trivially-single-step asks. (`worklog-review-2026-04`, Tier-1 #5)

- **Unprefixed 40-hex secret regex is over-eager today.** Modern secrets all use typed prefixes (`ghp_`, `sk-`, `AIza`, `AKIA`, `github_pat_`, `xox*`). Legacy GitHub PATs (the only realistic match) were deprecated in 2021. Drop unprefixed `\b[a-f0-9]{40}\b` patterns from secret scrubbers — they redact every git-SHA citation. (`5cd0b71`, H1)

- **YAML colons in prose values require quoting.** Any script that writes free-text frontmatter (e.g., `next_action`) must double-quote with backslash + quote escaping; otherwise a colon inside the prose ("Next: foo") triggers the "mapping values are not allowed here" parse error and the file becomes invisible to strict-YAML tooling. Same root cause across two independent failures (`marketing-intake-workflow.md`, `worklog-bug-fixes-2026-04.md`). (`af55a13`, B5)

- **`related[]` resolves to archived targets; only directional relations block archive.** `parent_slug`, `supersedes`, `reopens` imply structural dependence and should refuse archive without reparenting. `related[]` is a peer link and per AGENTS.md § Task relations resolves fine post-archive. The orphan-check should not refuse on `related[]` — that creates workflow friction without preventing real orphans. (`a3f7660`, B1)

- **Body-only cross-task references rot.** When a task body cites another slug, promote it to `related[]` immediately — not "later." Body mentions are grep-findable but not graphable; `bin/children.sh` and `bin/lint.sh --cross-task` only see frontmatter relations. The 2026-04-26 audit found 27 undeclared real refs across 8 active tasks — every one was a deliberate cross-reference that simply hadn't been promoted. (`a00adaa`, H4)

- **Verify perf/noise hypotheses with grep before fixing them.** Two recent instances were dissolved by a single search: the autosave LDAP-noise premise was speculative (no actual `resolve_ldap` call on the hook path), and the `checkpoint.sh → index.sh` hot-path hypothesis was wrong (only `archive.sh` calls it). Cheap to falsify; expensive to fix something that isn't broken. (`a00adaa`, H5)

- **Round-trip tests are the cheapest near-spec for markdown + frontmatter.** No real schema exists for these formats, so the closest thing to type-checking is "parse it, re-emit it, diff." Apply to export artifacts (clean corpus through scrub) and to frontmatter shapes (list-of-mappings via `yaml.safe_load`/`safe_dump`). Caught the list-of-mappings drop class before it could ship. (`62226e4`, worklog-self-improvement-2026-04 § C2)

- **Doc-code atomicity matters; doc drift is the dominant maintenance bug class in this repo.** Across 2026-04 sessions, ≥8 distinct fixes were pure doc-vs-code reconciliation (legacy `kind:`/`status:` values outside the documented set, helpers.md describing pre-B1 archive behavior, AGENTS.md missing `lint.sh`, modes/sync.md silent on lint integration, helpers.md claiming CI that didn't exist, etc.). Mitigation: when changing a script that AGENTS.md / protocol.md / helpers.md describes, edit the doc in the same commit. The `lint.sh --cross-task` mode catches some shapes (relation drift); structural drift between code and prose is still a human responsibility. (`a6bda48`, G1; `0cc7218`, G2; `82d7f4d`, B6)
